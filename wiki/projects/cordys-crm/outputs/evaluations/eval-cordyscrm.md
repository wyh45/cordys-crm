## 一、产品理念解读

CordysCRM 的产品定位（README.md 明确）:

> **Cordys CRM** 是新一代的开源 **AI CRM 系统**，集信息化、数字化、智能化于一体。

核心能力宣言:
- **AI 赋能 · 智能提效**: 开放 CRM Skills 接口，赋能 OpenClaw AI 销售助理 **7×24 小时智能服务**
- **智能 BI · 决策赋能**: 深度融合 DataEase 分析引擎，驱动精准决策与业绩增长
- **数字员工进化**: 不仅仅是工具，而是能自主判断、自动行动的智能体（Agent）

**健康管理模块**在 CordysCRM 中的定位（context.md）:
> 为医疗机构/体检中心提供**电子健康档案、AI 解读、健康干预（随访）、健康宣教（知识库+推送）** 四大能力。

## 二、符合产品理念的功能（✅）

### 1. Dify AI 健康解读 — 核心 AI 能力

**现状**: `HealthAiService` 对接 Dify，支持 5 种响应格式兜底解析，`/health/ai/interpret` 端点已实现。前端 AI 解读页面（`health/ai/index.vue`）有完整表单（档案选择 + 报告类型 + 验血指标 + 动态表单）+ 结果展示（解读/建议/警告）。

**符合理由**:
- 这是健康管理模块中最接近「AI工作台」理念的功能
- AI 解读直接将原始体检数据转化为可读健康建议
- 前端实现质量高：支持 GENERAL/BLOOD_TEST/URINE_TEST/ECG/X_RAY/CT/MRI 多种报告类型
- 一键复制结果，体验流畅

**对齐程度**: ⭐⭐⭐⭐（4/5）— 方向正确，但 AI 解读目前是**被动调用**（用户手动填数据触发），而非主动推送

---

### 2. 随访规则引擎 — 数字员工自动化基础

**现状**: `HealthFollowRuleService` 实现了 JSON 规则引擎，支持 AND/OR + 变量 + 操作符的条件匹配。随访记录 CRUD + 标记已联系完整。`HealthFollowRemindJob`（每日9:00执行）已设计但未部署。

**符合理由**:
- 规则引擎是「数字员工」自动执行任务的核心能力
- 自动触发随访（`HealthFollowRemindJob`）体现了「7×24小时智能服务」理念
- 随访日历视图（Task-C9 P2 阶段）可进一步强化可视化自动执行

**对齐程度**: ⭐⭐⭐（3/5）— 规则引擎骨架已有，但规则是**静态配置**（管理员手动定义），非 AI 动态生成规则

---

### 3. 体检数据同步 — 数据采集自动化

**现状**: `HealthSyncService` 实现两步同步（列表+详情），`workNo` 去重，`HealthSyncJob` 每日凌晨2:00自动执行，Redisson 分布式锁防止冲突。手动同步+定时同步双模式。

**符合理由**:
- 体现了「数字员工」的数据采集能力
- 增量同步（昨天整天）设计合理
- 分布式锁保证多实例安全

**对齐程度**: ⭐⭐⭐（3/5）— 数据拉取自动化，但仅限被动等待体检 API 数据，**无主动健康监测**

---

### 4. 推送功能（短信+内部通知）— 触达自动化

**现状**: `HealthPushService` 支持短信（腾讯云SMS）+ 内部通知（INSITE/EMAIL/DINGTALK/LARK/WECOM）双通道。`pushKnowledge()` 优先短信，失败降级内部通知。

**符合理由**:
- 自动化触达客户（「数字员工」主动联系客户）
- 多通道兜底确保触达率
- 推送记录可追踪（`health_push_record`）

**对齐程度**: ⭐⭐（2.5/5）— 有触达能力，但**推送内容是静态模板**，非 AI 个性化生成

---

## 三、偏离产品理念的功能（❌）

### 1. 健康知识库 — 完全偏离「AI工作台」

**问题**: `HealthKnowledge` 表 = 静态文章列表。知识库 CRUD（Task-C11 P2阶段）计划加富文本编辑器 + 分类 + 标签。

**为什么偏离**:
- CordysCRM 主产品已将 MaxKB（智能体平台）集成到生态
- 真正的「AI工作台」知识库应该是：**基于 RAG 的智能问答**，而非管理员手动编辑文章
- 当前设计是十年前的 CMS 思维，不是 AI 时代的知识服务
- 无个性化推送逻辑：同一个健康知识推送给所有相关客户，无千人千面

**建议**: 接入 MaxKB RAG 能力，或至少实现「基于客户体检指标的智能知识推荐」

---

### 2. 健康档案 — 仅数据存储，无「数字员工」主动性

**现状**: `HealthArchive` = 健康档案主表 + 子表（过敏史/病史/疫苗接种）。档案 CRUD 完整。

**为什么偏离**:
- 档案只存储数据，不产生任何**主动行动**
- 数字员工的标志是「感知→判断→行动」闭环。当前档案只有「感知」，无「判断」和「行动」
- 例如：体检报告显示血糖偏高 → 应自动生成随访任务 + 推送相关健康知识 + 标记高风险客户
- 当前这些全部是人工操作

**建议**: 在 `HealthArchive` 上增加「健康风险评分」字段，由 AI 自动计算并驱动后续行动

---

### 3. 异常指标分级 — 有统计无行动

**现状**: Task-C7b 后端已完成 `/health/examination/abnormal/stat` 端点，支持 NORMAL/CAUTION/WARNING/DANGER 四级分类，按检查项统计异常率。

**为什么偏离**:
- 「智能 BI · 决策赋能」不应该是统计展示，而应该是**主动预警**
- 当前只有 in-memory 聚合查询，医生需要自己去看报表发现异常
- 缺少：客户级风险预警（多项指标异常 → 高危）→ 自动生成随访任务

**建议**: 在统计基础上增加「行动触发」机制，异常率超阈值时自动生成待办任务

---

### 4. 随访规则 — 静态规则引擎，缺少 AI 推理

**现状**: `HealthFollowRule` = JSON 条件配置（年龄范围/BMI区间/指标阈值）。`HealthFollowRuleService` 根据规则模板匹配客户，生成随访任务。

**为什么偏离**:
- 规则是管理员手动配置的 if-then，不是 AI 学习的
- 缺少：基于历史随访数据的规则优化（哪些规则有效？哪些无效？）
- 缺少：自然语言配置规则的能力（"当客户50岁以上且血糖>6.1时，3个月内随访"）

**建议**: 引入 LLM 辅助生成随访规则，或基于历史成功案例推荐规则模板

---

### 5. 全模块缺失：AI 驱动的主动健康干预

**根本问题**: 当前健康管理模块是「**被动管理**」——等客户来体检→等医生手动解读→等随访到期通知。

**缺失的关键能力**:
1. **预测性干预**: 基于历史数据预测客户健康风险（如：连续3次体检血糖逐年上升→预警）
2. **个性化健康计划**: AI 根据客户档案自动生成个性化健康改善计划
3. **闭环反馈**: 随访完成后自动评估效果，下次体检时自动对比
4. **智能分群**: 基于健康指标对客户进行 AI 智能分群（高危/中危/低危/健康），差异化管理

## 四、优先级调整建议

### 当前优先级（P0~P3）的问题

| 任务 | 当前优先级 | 建议调整 | 理由 |
|------|-----------|---------|------|
| 档案 CRUD | P1 | 🔽 P3 | CRUD 是基础但非差异化价值 |
| 知识库 CRUD | P2 | 🔽 删除 / 🔺 P1（重构为 AI 知识推荐） | 静态知识库偏离产品理念 |
| AI 解读 | P1 (Task-C6/C7) | 🔺 最高优先级（P0） | 唯一真正体现「AI工作台」的核心功能 |
| 随访规则引擎 | P2 | 🔺 P1 | 数字员工自动化执行的核心 |
| 异常指标统计 | P1 (Task-C7b) | 🔺 P1 + 增加行动触发 | 统计+行动 = 真正的决策赋能 |
| 定时同步 | P1 (Task-C8) | 维持 | 数据采集自动化，合理 |
| 短信模板管理 | P3 | 🔽 维持P3 | 运营工具，非核心能力 |
| 体检报告导出PDF | P3 | 🔽 删除 | 非 AI 理念，与竞品同质化 |

### 重构后的优先级建议

```
P0（AI工作台核心）:
├── 完善 AI 解读 → 增加主动解读（档案变更时自动触发）
├── 基于异常的自动随访触发（统计超阈值 → 生成随访任务）

P1（数字员工基础能力）:
├── 智能健康分群（AI 评分 + 差异化管理策略）
├── 随访效果闭环追踪（对比前后指标）
├── 知识库重构为「智能知识推荐」（基于客户指标推荐知识）

P2（增强自动化）:
├── 预测性健康预警（趋势分析）
├── 个性化健康计划生成（AI + 规则引擎）
├── 推送内容 AI 个性化（不再是静态模板）

P3（运维支撑）:
├── 短信/推送通道管理
├── 数据导出
├── 随访日历可视化
```

## 五、综合评分

| 维度 | 得分 | 说明 |
|:---|:---:|:---|
| **魅力值** | 4/10 | 对医疗机构的吸引力一般。与现有体检系统（美年/爱康）相比无差异化优势 |
| **便捷性** | 5/10 | CRUD 操作流畅，但缺少一站式健康工作台体验 |
| **功能性** | 5/10 | 覆盖健康档案全生命周期，但 AI 能力仅停留在「解读」，未形成闭环 |
| **重要性** | 6/10 | 对医疗 CRM 场景有价值，但非 CordysCRM 主赛道的核心功能 |
| **实现复杂度** | 6/10 | 后端骨架已有，新增 AI 能力需要 Dify 应用侧配合 + 前端改造 |
| **风险** | 4/10 | 偏离产品理念的开发越多，后期重构成本越高 |
| **综合分** | **22** | **低优先级，需重构方向** |

**判决**: 不建议按当前方向继续开发。现有开发实现了「健康数据的 CRUD + 人工管理」模式，与 CordysCRM「AI工作台+数字员工」的产品理念存在根本性偏差。应优先重构为 AI 驱动的主动健康干预模式，再继续开发。

## 六、具体建议

### 建议 1: 健康管理模块应定位为「AI 健康助手」

不应只是档案管理工具，而应是:
> **「7×24小时智能健康管家」** — 自动监测客户健康状态，主动预警，智能干预

### 建议 2: 增加「健康风险评分」核心指标

```
健康风险评分 = AI(体检指标 + 历史趋势 + 家族史 + 随访记录)
```

这个评分驱动所有后续行动:
- 评分上升 → 自动生成高优先级随访
- 评分持续高危 → 触发预警通知
- 评分下降 → 推送鼓励知识

### 建议 3: 接入 CordysCRM 已有 AI 生态

CordysCRM 已有:
- MCP Server（智能创建/录入）
- MaxKB（企业知识库）
- SQLBot（智能问数）
- OpenClaw AI 助理

健康管理模块应与这些能力整合，而非独立造轮子。

### 建议 4: 将「知识库」重构为「智能健康顾问」

当前设计: 管理员写文章 → 推送给客户  
重构后: AI 分析客户健康数据 → 自动推荐最相关的健康知识/建议
